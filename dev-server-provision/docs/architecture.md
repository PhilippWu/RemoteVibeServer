## Component Descriptions

### 1. Cloud-Init Layer

The **cloud-init.example.yaml** file is the single entry-point for provisioning. It:

- Updates and installs system packages
- Writes the secure environment file to `/etc/dev-server/env` (mode 0600)
- Installs Docker from the official repository
- Configures UFW firewall (ports 22, 80, 443 only)
- Enables fail2ban for brute-force protection
- Executes the bootstrap script

Cloud-init is chosen because it is the **de facto standard** for VM initialization across all major cloud providers (AWS, GCP, Azure, Hetzner, DigitalOcean).

For servers without cloud-init support, the repository includes **install.sh** — a one-liner shell script that reads a pasted `RVSconfig.yml` (generated by the configurator), parses it into `/etc/dev-server/env`, and runs the same bootstrap→setup pipeline.

### 2. Bootstrap & Setup Scripts

**bootstrap.sh** is a thin wrapper that:
- Loads the environment file
- Downloads the full provisioning suite from GitHub
- Delegates to `setup.sh`

**setup.sh** is the main orchestrator:
1. Validates required environment variables
2. Detects the public IP via external APIs
3. Calls modular scripts in `infra/`
4. Installs and starts Coder
5. Writes deployment status artifacts

### 3. DNS Automation (infra/dns.sh)

Uses the **Cloudflare API v4** to create or update an A record.

**Why Cloudflare?**
- Industry-leading DNS with fast propagation
- Well-documented API with scoped API tokens
- Free tier is sufficient for development use
- DDoS protection as a bonus

### 4. Reverse Proxy (infra/proxy.sh)

**Caddy** is used instead of Nginx or Traefik.

| Criteria              | Caddy        | Nginx        | Traefik      |
|-----------------------|:------------:|:------------:|:------------:|
| Auto HTTPS            | ✅ Built-in  | ❌ Manual    | ✅ Built-in  |
| WebSocket support     | ✅ Native    | ✅ Config    | ✅ Native    |
| Config complexity     | Very low     | Medium       | Medium       |
| Binary size           | Single file  | Package      | Single file  |
| Certificate renewal   | Automatic    | Cron-based   | Automatic    |

**Decision:** Caddy wins on simplicity. For a single-service reverse proxy, its zero-config HTTPS is unbeatable.

### 5. Remote IDE Platform (Coder + code-server)

We evaluated three platforms:

| Feature                  | Coder (v2)        | code-server        | VS Code Server     |
|--------------------------|:-----------------:|:------------------:|:------------------:|
| VS Code Web              | ✅                | ✅                 | ✅ (preview)       |
| VS Code Desktop Remote   | ✅ (SSH)          | ❌                 | ✅                 |
| Extension marketplace    | ✅ Open VSX + MS  | ✅ Open VSX only   | ✅ Full MS         |
| Multi-user               | ✅                | ❌                 | ❌                 |
| Template system          | ✅ (Docker/K8s)   | ❌                 | ❌                 |
| Self-hosted              | ✅ OSS            | ✅ OSS             | ⚠️ Proprietary     |
| Auth & RBAC              | ✅                | Basic              | ❌                 |
| Active development       | ✅                | ✅                 | ⚠️ Limited         |

**Decision:** **Coder v2** is selected as the orchestration layer because it:
- Supports **both** VS Code Web and Desktop Remote via SSH
- Has a powerful **template system** for reproducible workspaces
- Is **open-source** (Apache 2.0) and self-hostable
- Provides **built-in authentication** and audit logging
- Manages **Docker-based workspaces** out of the box

**VS Code Web** is delivered by **[code-server](https://github.com/coder/code-server)** running inside each workspace container on port 13337. It appears as a single-click button ("VS Code Web") in the Coder dashboard — Coder proxies and authenticates the connection. Agents inside the workspace can read and edit the Coder workspace template via the Coder CLI and API.

### 6. Workspace Template (coder/main.tf)

AI coding agents are **optional** and controlled via environment flags:

```
ENABLE_AGENT_COPILOT=true   → installs GitHub Copilot CLI
ENABLE_AGENT_CLAUDE=true    → installs Anthropic Claude Code
ENABLE_AGENT_GEMINI=true    → installs Google Gemini CLI
ENABLE_AGENT_CODEX=true     → installs OpenAI Codex CLI
ENABLE_AGENT_OPENCODE=true  → installs OpenCode AI coding agent
```

OpenCode supports multiple upstream providers via `OPENCODE_PROVIDER`:

| Provider Value    | Description                              | Auth Method           |
|-------------------|------------------------------------------|-----------------------|
| `opencode-zen`    | OpenCode Zen (Recommended)               | GitHub OAuth          |
| `opencode-go`     | OpenCode Go — Low cost subscription      | GitHub OAuth          |
| `openai`          | OpenAI (ChatGPT Plus/Pro or API key)     | `OPENAI_API_KEY`      |
| `github-copilot`  | GitHub Copilot                           | `GITHUB_TOKEN`        |
| `anthropic`       | Anthropic (API key)                      | `ANTHROPIC_API_KEY`   |
| `google`          | Google                                   | `GOOGLE_API_KEY`      |

Each agent requires its corresponding API key. If the key is missing, the agent is skipped with a warning — the provisioning never fails due to a missing optional component.

### 8. Security Layers

- **UFW Firewall** — only ports 22, 80, 443 are open
- **fail2ban** — protects SSH from brute-force attacks
- **Caddy HTTPS** — automatic TLS with HSTS headers
- **Environment file** — stored at `/etc/dev-server/env` with mode 0600 (root-only)
- **No secrets in repository** — all credentials are injected at deploy-time via cloud-init

## Design Principles

1. **Single entry-point** — one cloud-init file deploys everything
2. **Idempotent** — every script is safe to re-run
3. **Modular** — each concern lives in its own script
4. **Fail-fast** — scripts use `set -euo pipefail` and validate inputs
5. **Observable** — all actions log to `/var/log/dev-server-provision.log`
6. **Secure by default** — minimal ports, encrypted transport, no secrets in code
